### 函数重载

- 主要就是通过函数名相同，但函数参数不同来实现的，c++能够支持但c不行
- 它的实现主要是基于函数名修饰来实现的，正常情况下，程序在链接时，链接器要去找到你调用的这个函数是在哪，比如a.o文件里用了add函数，但是链接器在该文件里找不到，就去b.o文件里找，首先，它会通过b.o的函数符号表里找，通过函数名找到它相对应的地址，那这时候的函数名的作用就体现出来了。
> 这里的表述不太对，应该说：每个 .o 文件里都有符号表（symbol table），符号表里存的是：符号名 + 它在文件里的偏移地址，链接器不是 “去别的文件里找地址”，而是：
    1. 本文件如果只有声明（没有实现），符号是未定义（undefined）
    2. 链接器去其他 .o 里查同名符号是否有定义
    3. 找到后把调用处的地址填进去

- c++对函数名的修饰通常会加上参数类型来进行修饰，这就使得它能够对函数名相同但参数不同的参数加以区分，也就是：函数名 + 参数个数 + 参数类型 + 顺序 生成唯一符号名，因而在函数符号表里找到对应的地址，而c语言没有这个函数名用参数类型修饰的机制，C 语言编译后，函数在符号表中的名字就是原函数名，不携带参数信息。只要函数名相同，符号就冲突，链接器无法区分，因此不支持重载。

- 扩展：extern "C"
    1. 用于告诉 C++ 编译器按 C 规则编译，不进行名字修饰。
    2. 常用于 C++ 与 C 代码互相调用。
    3. 被 extern "C" 修饰的函数不支持重载。

### 引用容易搞错的点


- 引用底层是用指针常量实现的，语法上我们认为它是不占空间的，但底层其实是占空间的，占的大小和指针一样

> int* p：普通指针，可以指向别的变量
> int* const p：指针本身是常量,指针变量 p 不能再指向别的变量但指向的内容可以改
> 指针可以不初始化，可以为空，可以野指针引用必须初始化，不能为空，不能重定向,语法上杜绝了很多错误
- 如果a是int类型，这时候const int& ra=a;是可以的，为啥呢，想想：常量引用的特性就是不能修改但能用，给你用只要你不要对我有修改的动作，这本质上是安全的。不过如果int& ra=a(a是const int)那就不行了，常量本来就不能被修改的，如果可以被转为可修改类型就完全乱套了
- const 引用可以绑定：普通变量 const 变量 临时变量 / 右值
> 左值：有地址有名字能在=左边；右值：没地址没名字在=右边(临时的，不能被修改，具有常性)
- 常引用的作用
    - 函数参数避免拷贝，提高效率
    - 同时保证不修改实参
    - 能接收临时变量、常量、表达式

- 引用作为返回值
    - 返回引用本质是返回变量的别名，不产生拷贝，效率高，且返回结果是左值，可以被赋值。
    - 核心原则：返回引用的变量，生命周期必须长于函数调用，函数结束后仍然存在。
- 可以安全返回引用的情况
    - 返回静态局部变量引用
    - 返回全局变量引用
    - 返回类成员变量引用
    - 返回 *this（用于运算符重载链式调用）
    - 返回传入函数的引用参数
- 绝对不能返回引用的情况
    - 返回普通局部变量引用（栈上，函数结束销毁，产生悬空引用）
    - 返回临时变量 / 表达式结果引用
    - 返回值类型函数的临时结果

### 内联函数

- 内联函数在编译时将函数体展开，避免函数调用的栈帧与跳转开销，以空间换时间。inline 只是建议，编译器可自行优化。短小高频函数适合内联，复杂函数、递归函数不适合。
- 优点
    1. 消除函数调用开销（压栈、跳转、返回）
    2. 提高运行速度
    3. 比宏 #define 更安全（有类型检查、不会诡异替换）
- 缺点
    1. 以空间换时间
    2. 函数被多次调用，代码会被复制多份 → 可执行文件变大
    3. 代码膨胀可能导致 CPU 缓存命中率下降，反而变慢

### 条件编译

目前为止主要用它的功能：

- 防止头文件重复包含（最常用）
```
#ifndef _HEADER_H
#define _HEADER_H
// ...
#endif
```
- 跨平台代码兼容
```
#ifdef _WIN32
    // Windows 平台代码
#elif __linux__
    // Linux 平台代码
#elif __APPLE__
    // macOS 代码
#else
    // 其他平台
#endif
```
- 功能裁剪，按宏控制是否编译某功能
```
#define USE_FEATURE_A

#ifdef USE_FEATURE_A
    // 功能A代码
#endif
```

### 内存对齐

- 为啥要内存对齐：因为cpu读取对齐的数据会更快，所以编译器会自动插空字节以达到内存对齐的规则

三条重要的对齐规则：
1. 每个基本类型的自身对齐值就是它的大小：
    - char → 1
    - short → 2
    - int → 4
    - float → 4
    - double → 8
    - 指针 → 4/8 字节（32/64 位）
2. 成员对齐规则
    - 结构体 / 类里每个成员变量相对于结构体起始位置的偏移量，必须是 min(自身大小, 对齐系数) 的整数倍。不满足就补空字节（padding）。
3. 结构体整体总大小规则
    - 整个结构体 / 对象的总大小必须是 min(结构体中最大成员大小, 对齐系数) 的整数倍。不满足就末尾再补 padding。

- c++类大小的计算
    1. 普通成员变量 → 参与大小计算，按内存对齐来
    2. 普通成员函数 → 不占对象大小（存在代码段）
    3. 静态成员变量 / 静态函数 → 不占对象大小（全局数据区）
    4. 有虚函数 → 对象多一个 虚指针（vfptr），占 4/8 字节
    5. 继承会叠加父类成员大小

### 赋值运算符重载

- 一般来说，简单的运算符重载我们都即可以写在类内或者类外，写在类内一般参数只需要写重载符号的右边的那个参数，这是因为this的存在，那写在类外有时得访问到类内私有变量，所以，有时可能得与友元配合
>现在尝试去重载operator<<，然后发现没办法将operator<<重载成成员函数。因为cout的
输出流对象和隐含的this指针在抢占第一个参数的位置。this指针默认是第一个参数也就是左操作
数了。但是实际使用中cout需要是第一个形参对象，才能正常使用。所以要将operator<<重载成
全局函数。但又会导致类外没办法访问成员，此时就需要友元来解决。operator>>同理。
- 不过，强调的是赋值运算符重载不能写在类外，因为如果写在类外，类里没写，就会有默认的赋值运算符重载函数，这时候就会与类外的重载冲突了

- 类的6个默认成员函数 
1. 构造函数 
2. 析构函数 
3. 拷贝构造函数 
4. 赋值运算符重载 
5. 移动构造函数（C++11）
6. 移动赋值运算符（C++11）

### const修饰成员函数

- 本质上它的作用就是把this指针变为const-->void show(const A* this) { ... },从而你只能访问不能修改，就会更加安全
- 一般来说，如果你已经明确了这个函数就是不要修改只用来访问的，并且它的返回值不是要来做左值的，那你自己写得这个函数就可以直接加上const，因为非const对象是可以调用const函数的
>之前在模拟实现STL容器的时候，有些函数经常既要写const也要写非const,主要就是因为返回值是左值还是右值的问题:为了让 const 对象能安全调用，同时让普通对象能修改。
```
// 非 const 版本
int& operator[](int i) {
    return arr[i];
}

// const 版本
const int& operator[](int i) const {
    return arr[i];
}
```
